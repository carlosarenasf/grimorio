# Grimorio v1 — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build Grimorio — a real-time D&D 5e (2024 / SRD 5.2) virtual tabletop with server-side role-based visibility, persistent accounts and campaigns, full 5e character sheets, and live combat — per `SPEC.md` and `DESIGN_SPEC.md`.

**Architecture:** Backend is hexagonal / clean architecture in TypeScript: a pure `domain` layer (no I/O), an `application` layer of command handlers, an `infra` layer of port adapters (Postgres + WS broadcaster), and a `transport` layer (Fastify HTTP + WebSocket). Server is authoritative for dice and all 5e derived stats. Role-based visibility is enforced by a single projection function in the domain before any state reaches a client. Frontend is React + Vite consuming HTTP for setup and WebSocket for live state. State is persisted as aggregate JSONB snapshots so any LiveTable can rehydrate after Render spin-down.

**Tech Stack:** TypeScript, pnpm workspaces, Vitest (TDD), Fastify + `@fastify/websocket`, `postgres` (postgres.js), `argon2`, `zod`, `nanoid`, ESLint + Prettier (backend); React + Vite + TypeScript (frontend); Postgres (Neon/Supabase).

---

## How to use this plan

This is a product-scale plan organised into **parallelization waves**. Within a wave, every task touches a **disjoint set of files** and has **no dependency on its siblings**, so each task is dispatched to its own subagent and they run concurrently. Between waves there is a hard dependency (a later wave imports types/ports defined earlier), so waves run in order.

Each task lists: its **owned files** (no other task in the same wave writes these), its **contract** (exact signatures/types it must export — these are frozen in Wave 0 so siblings can rely on them), and its **TDD test list** (the concrete behaviours to red-green-refactor). Every task is TDD: write the failing test, watch it fail, implement minimally, watch it pass, refactor, commit.

**Global rules for every task:**
- TDD always: no implementation line without a failing test first.
- One behaviour per test; tests are the spec. Name tests by behaviour, not by method.
- Pure domain code imports nothing from `infra`/`transport`/`node:*` (enforced by an ESLint boundary rule from Wave 0).
- Commit after each green+refactor cycle with a conventional-commit message.
- Run `pnpm -C backend test` (or `frontend`) before every commit; it must be green.

# Wave 1 — Pure domain modules (parallel ×8)

All Wave-1 modules are pure (no I/O), import only from `domain/types.ts`, `domain/ids.ts`, and (for randomness) receive an `Rng` argument — never call `Math.random` directly, so tests are deterministic with a seeded/fake `Rng`.

### Task 1a: Dice engine

**Owns:** `backend/src/domain/dice/parse.ts`, `roll.ts`, `index.ts`, `dice.test.ts`.

**Contract:**
```ts
parseNotation(s: string): { count: number; faces: number; mod: number } | null
rollNotation(notation: string, rng: Rng): { results: number[]; total: number; breakdown: string; tone: 'normal'|'crit'|'fumble' }
rollAttack(input: { toHitBonus: number|null; damage: string|null }, rng: Rng): { results: number[]; breakdown: string; total: number; tone: 'normal'|'crit'|'fumble' }
rollWithAdvantage(mod: number, mode: 'advantage'|'disadvantage'|'normal', rng: Rng): {...}
```
**TDD test list (write each as a failing test first):**
- `parseNotation` accepts `2d6+3`, `1d20`, `d8`, `1d20-1`, whitespace-tolerant; rejects `''`, `2x6`, `d1`, `0d6`, `999d20` (limit), faces ∉ {4,6,8,10,12,20,100}+sane range.
- `rollNotation('1d20', fakeRng→20)` → tone `crit`; →1 → `fumble`; →10 → `normal`.
- `rollNotation('2d6+3', fakeRng)` → results length 2, total = sum+3, breakdown string matches `a + b + 3`.
- `rollAttack({toHitBonus:7,damage:'1d8+4'})` → combined breakdown has hit and damage parts; nat 20 → crit.
- `rollWithAdvantage(5,'advantage',rng→[12,18])` keeps 18; disadvantage keeps 12.
- Commit per behaviour group.

### Task 1b: 5e rules calculator

**Owns:** `backend/src/domain/rules/abilities.ts`, `proficiency.ts`, `pointbuy.ts`, `derived.ts`, `index.ts`, `rules.test.ts`.

**Contract:**
```ts
abilityMod(score: number): number                  // floor((score-10)/2)
proficiencyBonus(level: number): number             // ceil(level/4)+1
pointBuyCost(score: number): number                 // SRD 5.2 table 8..15
pointBuyTotal(scores: Record<AbilityKey, number>): number
isLegalPointBuy(scores): boolean                     // total <= 27 and each 8..15
skillModifier(scores, ability: AbilityKey, proficient: boolean, level): number
spellSaveDc(scores, ability: AbilityKey, level): number  // 8 + prof + mod
initiative(scores): number                           // dex mod
```
**TDD test list:**
- `abilityMod(10)=0, abilityMod(18)=4, abilityMod(3)=-4`.
- `proficiencyBonus(1)=2, (4)=2, (5)=3, (20)=6`.
- `pointBuyCost(8)=0,(13)=5,(14)=7,(15)=9`; `isLegalPointBuy` true at exactly 27, false at 28, false if any score 16.
- `skillModifier` adds prof bonus only when proficient.
- `spellSaveDc` for level 5, cha 16 = 8+3+3 = 14.
- Commit per group.

### Task 1c: Visibility projection

**Owns:** `backend/src/domain/visibility/project.ts`, `statusLabel.ts`, `index.ts`, `visibility.test.ts`.

**Contract:**
```ts
hpStatusLabel(currentHp: number, maxHp: number): 'Intacto'|'Herido'|'Malherido'|'Caído'
projectLiveTable(table: LiveTable, viewer: { userId: UserId; role: Role }): MasterSnapshot | PlayerSnapshot
projectCombatant(c: Combatant, viewer): PublicCombatantDTO   // hides dm_only hp → statusLabel
```
**TDD test list (this is the product's core guarantee — test hard):**
- `hpStatusLabel`: full→Intacto, ≤66%→Herido, ≤33%→Malherido, ≤0→Caído.
- A player snapshot **never contains** `dmNotes`, hidden rolls, or numeric HP of a `dm_only` combatant; it contains the derived `statusLabel` instead.
- A monster with `hpVisibility:'dm_only'`: master snapshot has real `currentHp`; player snapshot has `statusLabel` and no `currentHp`.
- A PC combatant (`hpVisibility:'public'`): both snapshots show numeric HP.
- `dm_only` `DiceRoll` appears in master roll log, absent from player roll log.
- Property test: for any table, the player snapshot serialized to JSON contains no substring of any `dmNotes` value. Commit per group.

### Task 2d: LiveTable application + tests (the big one)

**Tests:** two simulated clients (dm + player) in a room; dm `RollHidden` is delivered to dm socket only; `NextTurn` broadcasts a `TurnAdvanced` to both with correct per-role projection; reconnecting client receives a fresh snapshot. Commit per behaviour.
